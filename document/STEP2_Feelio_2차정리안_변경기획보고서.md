## 1. 프로젝트 변경 배경

Feelio의 초기 기획은 "감정 때문에 새는 돈을 줄여주는 AI 소비 관리 서비스"였다. 감정소비 누수율이라는 수치 지표를 중심에 두고, 사용자의 소비를 분석해 절약을 유도하는 관리형 서비스에 가까웠다.

1개월 팀 프로젝트로 구현을 진행하면서 다음과 같은 상황 변화가 있었다.

- 팀의 실제 구현 속도와 범위를 고려할 때, 백엔드(인증·태그 시스템·분석 API)까지 포함한 초기 기획 전체를 기간 내에 완성하기 어려웠다.
- 화면을 만들어 가는 과정에서, 수치로 소비를 평가받는 경험보다 **감정을 캐릭터(말랑이)와 시각화(감정 능선, 감정 캘린더)로 되돌아보는 경험**이 Feelio의 정체성에 더 잘 맞는다는 방향이 잡혔다.
- 이에 따라 서비스의 무게중심이 "관리·절약"에서 **"기록·회고"**로 이동했고, 현재 프로젝트는 프론트엔드 중심으로 핵심 사용 경험을 먼저 완성하는 전략을 택했다.

## 2. 기존 기획의 문제점 또는 한계

| 구분 | 내용 |
|---|---|
| 범위 과다 | 회원가입/JWT 인증, 커스텀 태그 CRUD, 소비 기록, 캘린더, 목표, 분석, AI 피드백을 1개월에 풀스택으로 구현하는 계획은 팀 규모 대비 과도했다 |
| 지표 중심의 부담 | 감정소비 누수율은 "내 소비가 잘못됐다"는 평가로 받아들여질 수 있어, 가볍고 감성적인 기록 경험이라는 목표와 충돌했다 (해당 지표는 제거 확정) |
| 태그 시스템의 복잡도 | 감정·상황·분류 3종 태그 + 사용자 커스텀 CRUD는 백엔드 없이는 의미가 약했고, 감정 캐릭터 디자인과도 맞지 않았다 |
| 지출 전용 구조 | 지출만 기록하는 구조로는 한 달의 흐름(수입 대비 소비)을 온전히 돌아보기 어려웠다 |
| 홈 화면 과밀 | 홈 안에서 기록 카드가 확장되는 방식은 기록·요약·목표가 한 화면에 몰려 각 기능의 밀도가 떨어졌다 |

## 3. 강사님 피드백 반영 내용

> **추가 확인 필요:** 강사님 피드백의 구체적인 내용은 제공된 자료(초기 기획서, 코드, 작업 이력)에서 확인되지 않는다. 아래는 초기 기획 → 현재 프로젝트의 변경 결과로부터 역추정한 항목이며, 실제 피드백 내용을 전달받으면 이 절을 갱신한다.

| 반영된 것으로 추정되는 방향 | 현재 프로젝트에서의 반영 결과 | 근거 |
|---|---|---|
| 서비스 정체성을 명확히 할 것 | "감정 기반 소비 기록 서비스"로 정의 정리, 회계·관리 기능 축소 | 추정 |
| 핵심 기능에 집중할 것 | 기록 → 조회 → 회고 흐름을 우선 완성, 누수율·커스텀 태그 등 제거/후순위화 | 추정 |
| 차별화 요소를 살릴 것 | 감정 말랑이·감정 능선·평행우주 등 감정 시각화 요소를 전면 배치 | 추정 |

## 4. 현재 프로젝트의 최종 방향

- Feelio는 기록에서 출발해 **감정에 따른 소비 패턴 분석 인사이트**를 돌려주는 감정 소비 분석 서비스다. 무겁고 복잡한 회계 도구가 아니라, 가볍고 감성적인 기록·회고 경험 위에 인사이트를 얹는다.
- 사용자가 하는 일은 단순하다: **수입·지출을 기록하고, 그 순간의 감정을 붙이고, 나중에 돌아본다.**
- 서비스가 하는 일은 그 기록을 **감정의 언어로 되돌려주는 것**이다.
  - 오늘의 대표 감정 말랑이 (홈)
  - 한 달 감정의 흐름 — 감정 능선, 감정 캘린더 (홈)
  - 일별·월별·감정별 소비 목록 (거래내역)
  - 감정 소비 패턴 인사이트 (AI 분석)
  - 소비 습관이 만드는 두 가지 미래 (평행우주)
- 구현 전략: **프론트엔드로 핵심 경험을 먼저 완성**하고(현재 완료 단계), 백엔드(Spring Boot + MySQL + MyBatis)와 실데이터 분석은 후속 단계로 연결한다.

## 5. 핵심 서비스 정의

> Feelio는 사용자가 수입·지출 내역을 기록하고 각 소비에 감정을 함께 남기면, **감정에 따른 소비 패턴 분석 인사이트**를 제공하는 감정 소비 분석 서비스이다. 사용자는 이를 통해 자신의 소비 흐름을 감정의 관점에서 돌아볼 수 있다.

- **Feel + I/O**: 감정을 입력(Input)하면, 소비 인사이트가 출력(Output)된다.
- 8가지 기본 감정: 신남, 설렘, 뿌듯함, 스트레스, 외로움, 화남, 평온, 무덤덤 — 각 감정은 고유 색상과 말랑이 캐릭터로 표현된다.
- 기록의 최소 단위: 금액 + 감정(필수) + 카테고리(필수) + 상황·메모(선택) + 일시.

## 6. 주요 기능 방향

| 기능 영역 | 방향 | 구현 상태 |
|---|---|---|
| 소셜 로그인 | Google/Kakao/Naver 계정으로 가입 절차 없이 진입 | 현재 화면 기준 (mock 동작, 실제 인증 추가 구현 필요) |
| 온보딩 | 첫 로그인 시 5단계로 목표(이름·금액·기간·현재 금액)를 설정하고 시작 | 현재 코드 기준 구현 |
| 감정 소비 기록 | 지출/수입 선택 → 금액 → 감정 말랑이 선택(필수) → 카테고리(필수) → 상황·메모(선택) → 저장 | 현재 코드 기준 구현 |
| 홈 회고 | 선택한 날의 대표 감정 말랑이, 월간 감정 능선, 감정 색 미니 캘린더, 목표 진행률, AI 감정 신호 | 현재 코드 기준 구현 (AI 신호 문구는 하드코딩) |
| 거래내역 조회 | 일별/월별/감정별 그룹 뷰, 검색, 연도·월일·카테고리·감정 필터, 정렬, 그룹 합계 | 현재 코드 기준 구현 |
| 기록 관리 | 거래 상세 조회·수정·삭제 | 현재 코드 기준 구현 |
| AI 분석 | 감정-소비 상관 인사이트, 소비 위험도, AI 맞춤 챌린지, 카테고리별 예산 현황, 소비 코어(카테고리/시간/감정) 차트, 시간대 패턴 | 현재 화면 기준 (데이터 하드코딩, 실데이터 연동 추가 구현 필요) |
| 평행우주 | 현재 소비를 유지한 미래 vs 감정소비를 줄인 미래를 비교 시뮬레이션으로 보여주고 목표와 연결 | 현재 코드 기준 구현 (시나리오 데이터 고정) |
| 개인화 | 프로필 수정, 목표 관리, 다크/라이트 모드, 오로라 테마, 데이터 초기화 | 현재 코드 기준 구현 |

## 7. 화면 구조 변경 사항

### 초기 기획 (4개 탭 + 홈 내 기록 카드)

```text
로그인/회원가입 → 홈(기록 카드 확장) / 캘린더 / 콘텐츠(분석) / 목표
```

### 현재 프로젝트 (5개 탭 + 온보딩 + 프로필 모달)

```text
소셜 로그인 → 온보딩(목표 설정)
→ 홈 / 지출·수입 기록 / 거래내역 / AI 분석 / 평행우주
   (공통: 프로필 모달 - 프로필·목표·설정)
```

| 변경 항목 | 초기 기획 | 현재 프로젝트 | 의도 |
|---|---|---|---|
| 기록 입력 | 홈 안의 확장 카드 | 독립된 "지출·수입 기록" 탭 | 기록을 하루의 의식(ritual)처럼 온전한 화면으로 승격 |
| 캘린더 | 독립 탭 | 홈 우측 미니 감정 캘린더로 통합, 날짜 선택 시 거래내역과 연결 | 홈을 "회고의 중심"으로 만들고 탭 수 재배분 |
| 콘텐츠/분석 | 콘텐츠 탭 하나 | AI 분석 + 평행우주 두 개 탭으로 분리 | 데이터 인사이트와 스토리텔링형 콘텐츠의 역할 분리 |
| 목표 | 목표 화면/홈 카드 | 온보딩에서 최초 설정 + 홈 카드 + 프로필 모달에서 관리, 홈 목표 카드는 평행우주로 연결 | 목표를 별도 화면이 아닌 서비스 전반의 맥락으로 배치 |
| 회원가입 | 이메일 가입 화면 | 소셜 로그인 단일 화면 | 진입 장벽 최소화 |

## 10. 향후 보완 방향

1. **백엔드 구축**: Spring Boot + MySQL + MyBatis 기반으로 인증(소셜 로그인), 거래 기록, 목표 API를 구현하고 localStorage 상태를 서버 데이터로 대체한다.
2. **분석 실데이터화**: AI 분석 화면의 하드코딩 수치를 실제 기록 기반 집계(감정별·카테고리별·시간대별)로 교체한다.
3. **평행우주 시뮬레이션 로직 연결**: 고정 시나리오를 사용자 기록·목표 기반 계산으로 대체한다 (`planetScoring` 유틸 활용 검토).
4. **잔여 정리 작업**: 분석 화면의 누수율 UI 제거, 로그인 화면 소개 문구를 서비스 표현 규칙에 맞게 수정, 상황 태그 복수 저장 정책 확정, 데모용 고정 날짜(2026-07-01) 처리.
5. **AI 문장 생성 고도화**: 홈 감정 신호·분석 인사이트 문구를 룰 기반 생성으로 시작해 LLM API 연동으로 확장한다.
6. **강사님 피드백 반영 내역 보강**: 실제 피드백 내용을 확인해 3절을 확정한다.
