## 1. 초기 기획의 핵심 요약 (초기 기획 기준)

- Feelio는 사용자가 돈을 쓴 순간의 감정을 태깅하고, AI가 반복되는 감정 소비 패턴을 분석해 **감정 때문에 새는 돈을 줄이고 목표 달성을 돕는 AI 소비 관리 서비스**로 정의되었다.
- 핵심 컨셉: "감정을 입력하고, 소비 인사이트를 출력한다."
- 1개월 웹 MVP 목표로, 다음을 포함했다.
  - 이메일·비밀번호 **회원가입/로그인 + JWT 인증** //oauth
  - **태그 시스템**: 감정(EMOTION)·상황(SITUATION)·소비 분류(CATEGORY) 3종 + **사용자 커스텀 태그 CRUD·색상 관리**
  - **지출 중심 소비 기록** CRUD (홈 화면 안의 "기록 카드 확장 UI"에서 입력)
  - 홈: 감정소비 누수율, AI 한 줄 피드백, 최근 기록, 목표 진행률
  - 캘린더: 날짜별 대표 감정·총액
  - 목표: 생성/조회/진행률, 참은소비(SAVED) 반영 구조 
  - 분석: 누수율 계산, 감정별/분류별/시간대별 집계, AI 피드백
- 기술 구조: **Spring Boot + MySQL + MyBatis 백엔드**, React + Vite 프론트엔드, 7개 테이블 ERD(users, tags, expenses, expense_tags, goals, ai_feedbacks, monthly_analyses)와 REST API 명세까지 정의됨.
- **평행우주 콘텐츠, 친구 챌린지, MyData 연동 등은 MVP 제외(후순위)**로 명시되었다.

## 2. 현재 프로젝트의 핵심 요약 (현재 코드 기준)

- 현재 구현체는 **Vite + React + Emotion(CSS-in-JS) 기반 프론트엔드 단독 애플리케이션**이다. 백엔드·DB·API 코드는 저장소에 존재하지 않으며, 상태는 `localStorage`(`src/stores/useFeelioStoreDc.js`)와 목데이터(`src/data/mockTransactions.js`, `src/data/mockGoals.js`)로 관리된다.
- 서비스 성격이 **감정 기반 소비 기록 서비스**로 정리되었다. 사용자는 수입·지출을 기록하고, 각 기록에 감정을 붙이고, 일별·월별·감정별로 돌아본다.
- 화면 구성(5개 탭 + 로그인/온보딩/프로필, `src/app/routes.js` 기준):

| 화면 | 파일 | 핵심 내용 |
|---|---|---|
| 로그인 | `src/pages/LoginPage.jsx` | Google/Kakao/Naver 소셜 로그인(현재 mock 동작) |
| 온보딩 | `src/pages/OnboardingPage.jsx` | 5단계 초기 목표 설정(목표·금액·기간·현재 금액) |
| 홈 | `src/pages/HomePageDesign.jsx` | 감정 말랑이(대표 감정 캐릭터), 감정 능선(월간 감정 분포), 미니 감정 캘린더, 목표 카드, AI 감정 신호 |
| 지출·수입 기록 | `src/pages/RecordPageDc.jsx` | 지출/수입 토글, 금액, 감정 8종 선택(필수), 카테고리(필수), 상황(선택·복수), 메모, 일시 |
| 거래내역 | `src/pages/TransactionsPageDesign.jsx` | 일별/월별/감정별 뷰, 검색, 연도·월일·정렬·카테고리·감정 필터, 그룹 합계 |
| 거래 상세 | `src/components/transactions/TransactionDetailModal.jsx` | 상세 조회·수정·삭제 |
| AI 분석 | `src/pages/AnalysisPageDc.jsx` | 위험 감정 루트, 소비 위험도, AI 챌린지, 예산 현황, 소비 코어 차트, 시간대 패턴 (현재 전부 하드코딩 데이터) |
| 평행우주 | `src/pages/UniversePageDc.jsx` | "지금처럼 소비한 나" vs "감정소비를 줄인 나" 미래 비교 시뮬레이션 |
| 프로필 모달 | `src/components/profile/ProfileModalDc.jsx` | 프로필 변경, 목표 관리, 알림 설정, 다크모드·오로라 테마, 데이터 초기화, 로그아웃/회원탈퇴 |

- 감정은 테이블·커스텀 태그가 아니라 **고정 8종 문자열**(신남, 설렘, 뿌듯함, 스트레스, 외로움, 화남, 평온, 무덤덤 — `src/data/emotions.js`)로 처리되며, 카테고리·상황도 고정 목록이다.
- 거래 데이터 구조(현재 코드 기준): `{ id, type(expense|income), amount, category, emotion, situation, memo, date }` — 태그 ID 조인 방식이 아닌 문자열 필드 방식.

## 3. 초기 기획과 현재 프로젝트의 공통점

| 항목 | 내용 |
|---|---|
| 핵심 가치 | 소비 순간의 감정을 함께 기록한다는 방향은 동일 |
| 기록 3요소 | 금액 + 감정(필수) + 카테고리(필수), 상황·메모는 보조 — 초기 기획의 필수 입력 정책이 코드에 그대로 반영됨 (`RecordPageDc.jsx`의 `canSave = amount && emotion && category`) |
| 캘린더 회고 | 날짜별 대표 감정을 색으로 보여주는 월간 캘린더 개념 유지 (홈 내 미니 캘린더로 위치 변경) |
| 목표 연결 | 목표 금액·진행률을 만들고 소비 흐름과 연결하는 구조 유지 |
| AI 피드백 | AI가 감정 소비 패턴에 대해 코멘트하는 개념 유지 (현재 문구 하드코딩) |
| 기술 방향 | React + Vite 프론트엔드 구조 유지 |

## 8. 아직 구현되지 않았거나 불명확한 부분

## 9. 변경이 발생한 이유 추정

아래는 코드·작업 이력(CODEX_HANDOFF.md)에서 유추한 **추정**이다.

1. **1개월 학습 프로젝트 범위 조정**: 백엔드+태그 시스템+AI를 모두 만드는 대신, 사용자 경험(기록 → 회고 → 인사이트)을 먼저 프론트엔드로 완성해 시연 가능한 형태를 만드는 전략으로 보임.
2. **차별화 포인트의 이동**: 누수율 같은 수치 지표로 절약을 압박하는 관리형 컨셉보다, 말랑이·감정 능선·평행우주 같은 **감성적 시각화**가 발표·시연에서 더 강한 인상을 준다고 판단한 것으로 추정. 누수율 제거 결정도 이 방향과 일치한다.
3. **커스텀 태그의 후순위화**: 태그 CRUD는 백엔드 없이는 의미가 약하고, 고정 8감정이 캐릭터 디자인·색상 체계와 1:1로 묶이면서 자연스럽게 고정 목록이 된 것으로 추정.
4. **수입 기록 추가**: 소비 회고가 목적이라도 월급 등 수입 맥락이 있어야 소비 흐름이 완성된다는 판단으로 추정 (deep-research-report.md의 수입·지출 표준 구성 리서치와 부합).
5. **소셜 로그인 전환**: 데모 진입 장벽을 낮추고 회원가입 폼 구현 비용을 줄이기 위한 선택으로 추정.

## 10. 앞으로 문서화할 때 기준으로 삼아야 할 현재 프로젝트 방향

이후 STEP 2~8 문서는 아래를 기준으로 작성한다.

1. **서비스 정의**: "Feelio는 사용자가 수입·지출 내역을 기록하고 각 소비에 감정을 함께 남기면, **감정에 따른 소비 패턴 분석 인사이트**를 제공하는 감정 소비 분석 서비스이다."
2. **화면 축**: 로그인 → 온보딩(목표 설정) → 홈 / 지출·수입 기록 / 거래내역 / AI 분석 / 평행우주 + 프로필 모달.
3. **감정 모델**: 고정 8감정(신남, 설렘, 뿌듯함, 스트레스, 외로움, 화남, 평온, 무덤덤) 문자열 기준. ERD·DB 설계 시 "테이블 분리 vs enum/문자열"은 현재 코드 사실(문자열)을 기준으로 쓰고, 개선안은 "추가 설계 제안"으로 분리.
4. **제거 확정 기능**: **감정소비 누수율(계산·표시 모두)은 제거 확정**이므로 이후 요구사항 정의서·기능 명세서·API 명세서에 포함하지 않는다. 분석 화면에 남은 누수율 UI는 제거 대상 잔여물로만 언급한다.
5. **구현 상태 표기**: 프론트엔드 동작(기록 CRUD, 조회/필터, 캘린더, 목표, 테마)은 "현재 코드 기준 구현", 백엔드·인증·AI 연동은 "추가 구현 필요"로 일관 표기.
6. **초기 기획 산출물의 재활용**: API 명세 초안, ERD 초안은 "초기 기획 기준" 참고 자료로 인용하되, 현재 데이터 구조(수입 포함, 문자열 감정)에 맞게 갱신해서 사용. 누수율 공식은 재활용하지 않는다.
